    11	Non-sensitive categories (history, bookmarks, downloads, extensions, storage) are stored in plaintext and do not require decryption.
    12	
    13	## 2. Cipher Version Detection
    14	
    15	Every encrypted value begins with a 3-byte prefix that identifies the cipher version:
    16	
    17	| Prefix | Version | Meaning |
    18	|--------|---------|---------|
    19	| `v10` | CipherV10 | Chrome 80+ standard encryption (AES-GCM on Windows, AES-CBC on macOS/Linux) |
    20	| `v20` | CipherV20 | Chrome 127+ App-Bound Encryption |
    21	| (none) | CipherDPAPI | Pre-Chrome 80 raw DPAPI encryption (Windows only, no prefix) |
    22	
    23	If the ciphertext is shorter than 3 bytes or the prefix is unrecognized, it is treated as legacy DPAPI.
    24	
    25	## 3. macOS Encryption
    26	
    27	Chromium on macOS stores a per-browser secret in the macOS Keychain (e.g. "Chrome Safe Storage", "Brave Safe Storage"). The master key is derived from this secret via PBKDF2:
    28	
    29	| Parameter | Value |
    30	|-----------|-------|
    31	| Hash | SHA-1 |
    32	| Salt | `saltysalt` |
    33	| Iterations | 1003 |
    34	| Key length | 16 bytes (AES-128) |
    35	
    36	Decryption uses AES-128-CBC with a fixed IV of 16 space bytes (`0x20`). The ciphertext layout:
    37	
    38	```
    39	| v10   | AES-CBC ciphertext (PKCS5 padded) |
    40	|-------|-------------------------------------|
    41	| 3B    | remaining bytes                     |
    42	```
    43	
    44	There are three retrieval strategies, tried in order: (1) gcoredump exploit for securityd process memory, (2) direct keychain unlock with user's login password, (3) `security` CLI command (may trigger a GUI prompt). See [RFC-006](006-key-retrieval-mechanisms.md) for details.
    45	
    46	## 4. Windows Encryption
    47	
    48	Chromium on Windows stores a base64-encoded encrypted key in `Local State` at `os_crypt.encrypted_key`. The key recovery process is:
    49	
    50	1. Base64-decode the `encrypted_key` value
    51	2. Strip the 5-byte `DPAPI` ASCII prefix
    52	3. Decrypt via Windows `CryptUnprotectData` (DPAPI) to obtain the 256-bit master key
    53	
    54	With the master key, each encrypted value is decrypted as AES-256-GCM:
    55	
    56	```
    57	| v10   | nonce  | ciphertext + auth tag (16B) |
    58	|-------|--------|-----------------------------|
    59	| 3B    | 12B    | remaining bytes             |
    60	```
    61	
    62	**Legacy DPAPI** — values without a `v10`/`v20` prefix (pre-Chrome 80) are passed directly to `CryptUnprotectData`:
    63	
    64	```
    65	| DPAPI blob (no prefix)             |
    66	|-------------------------------------|
    67	| variable length                     |
    68	```
    69	
    70	## 5. Linux Encryption
    71	
    72	Chromium on Linux retrieves a per-browser secret from D-Bus Secret Service (GNOME Keyring or KDE Wallet). The label matches the browser's storage name (e.g. "Chrome Safe Storage", "Chromium Safe Storage"). If D-Bus is unavailable, the hardcoded fallback password `peanuts` is used.
    73	
    74	The master key is derived via PBKDF2 with different parameters than macOS:
    75	
    76	| Parameter | Value |
    77	|-----------|-------|
    78	| Hash | SHA-1 |
    79	| Salt | `saltysalt` |
    80	| Iterations | 1 |
    81	| Key length | 16 bytes (AES-128) |
    82	
    83	Decryption uses the same AES-128-CBC scheme as macOS (fixed IV of 16 space bytes, PKCS5 padding).
    84	
    85	## 6. v20 App-Bound Encryption (Chrome 127+)
    86	
    87	Chrome 127 introduced App-Bound Encryption on Windows, identified by the `v20` prefix. This scheme binds the encryption key to the Chrome application identity, making it harder for external tools to decrypt. After decryption, the payload contains a 32-byte application header before the actual plaintext:
    88	
    89	```
    90	| v20   | nonce  | AES-GCM payload                    |
    91	|-------|--------|-------------------------------------|
    92	| 3B    | 12B    | remaining bytes                     |
    93	
    94	After decryption:
    95	| app-bound header | plaintext                          |
    96	|------------------|------------------------------------|
    97	| 32B              | remaining bytes                    |
    98	```
    99	
   100	**Current status**: v20 decryption is not yet implemented. Encountering a `v20`-prefixed value returns an error. This primarily affects recent Chrome installations on Windows.
   101	
   102	## 7. Decryption Flow
   103	
   104	The high-level decryption path for any encrypted Chromium value:
   105	
   106	1. **Detect version** -- inspect the first 3 bytes of the ciphertext
   107	2. **Route by version**:
   108	   - `v10` -- strip prefix, call platform-specific decryption (AES-CBC on macOS/Linux, AES-GCM on Windows)
   109	   - `v20` -- not yet supported, return error
   110	   - DPAPI (no prefix) -- call Windows `CryptUnprotectData` directly (Windows only; returns error on other platforms)
   111	3. **Return plaintext** -- the decrypted bytes are interpreted as a UTF-8 string
   112	
   113	Each record is decrypted independently. A failure to decrypt one value does not prevent extraction of other records in the same database.
